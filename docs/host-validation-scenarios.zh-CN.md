# 宿主验证场景

## 目的

本文档用于定义 AgentSkeleton 在 `Codex` 与 `Claude Code` 中的共享验证场景。

它的目标是验证：官方宿主接入产物是否真正交付了 `v0.1.0` 所要求的零命令工作流体验。

## 验证目标

当一个宿主接入能够在不要求用户手工管理 CLI 命令的前提下，稳定提供预期的 AgentSkeleton 工作流行为时，它才算通过验证。

验证重点应放在：

- 面向用户的零命令体验
- 正确的宿主动作路由
- 正确消费稳定协议字段
- 上下文变化后的继续推进能力
- 在应暂停自动化时做出正确升级

## 范围

这些场景应同时用于：

- 官方 Codex 接入
- 官方 Claude Code 接入

即使宿主执行环境不同，场景意图也应保持一致。

## 验证规则

在验证过程中：

- 除非场景明确要求，否则不要通过显式说出 AgentSkeleton 命令来帮助宿主
- 重点评估宿主能否自己选择正确动作
- 除用户显式要求外，向用户暴露命令名应被视为产品回归
- 验证依据应是共享宿主规范，而不是宿主侧临场发挥

## 场景 1：新项目初始化

### 目标

验证宿主是否能激活 AgentSkeleton，并在明显的新项目场景下选择 `init-docs`。

### 前置条件

- 仓库尚不存在 AgentSkeleton 上下文
- 仓库为空或接近空
- 用户要求帮助启动一个带 AI-friendly 文档结构的新项目

### 预期宿主行为

- 识别出当前应启用 AgentSkeleton
- 无需让用户选命令即可选择 `init-docs`
- 建立上下文
- 执行 `plan`
- 只有在起草被阻塞时才提出下一个必要澄清问题

### 失败信号

- 让用户选择要运行哪个 CLI 命令
- 误选 `reshape-docs`
- 一直停留在宽泛探索式提问，而不进入工作流

## 场景 2：老项目重塑

### 目标

验证宿主是否能激活 AgentSkeleton，并在现有仓库场景下选择 `reshape-docs`。

### 前置条件

- 仓库已有代码和文档，但没有 AgentSkeleton 上下文
- 用户要求把仓库整理成适合 agent 协作的文档结构

### 预期宿主行为

- 识别出当前应启用 AgentSkeleton
- 选择 `reshape-docs`
- 建立上下文
- 执行 `plan`
- 根据返回的工作流状态进入澄清或起草

### 失败信号

- 无充分理由地选择 `init-docs`
- 让用户手工检查或创建上下文文件
- 向用户暴露命令级工作流

## 场景 3：继续当前 Priority 起草

### 目标

验证宿主是否使用 `focus-doc` 起草，而不是只看 `plan` 就直接发挥。

### 前置条件

- 仓库已存在 AgentSkeleton 上下文
- `plan` 已给出明确的 `current_priority`
- required context 足以支持起草
- 用户要求继续文档工作

### 预期宿主行为

- 执行 `plan` 或使用仍然新鲜的工作流状态
- 选择当前 priority 文档
- 执行 `focus-doc`
- 基于返回 package 起草

### 失败信号

- 不经过 `focus-doc` 就直接从 `plan` 起草
- 当 `current_priority` 已明确时，仍让用户决定下一份文档
- 把 drafting package 内部字段直接暴露给用户，而不是转化成面向任务的进度

## 场景 4：起草前的阻塞澄清

### 目标

验证当起草尚不可靠时，宿主是否只追问最小必要问题。

### 前置条件

- 仓库已存在 AgentSkeleton 上下文
- 当前 priority 因缺少 required context 而未 ready
- 用户要求继续或起草下一份文档

### 预期宿主行为

- 识别阻塞性的 missing context
- 只追问一个高价值问题
- 避免宽泛探索式提问
- 除非显式允许占位起草，否则不应带着缺失事实硬写

### 失败信号

- 一次抛出多个不聚焦问题
- 在编造事实的前提下继续起草
- 让用户自己判断下一步命令

## 场景 5：写回结构化答案并继续推进

### 目标

验证宿主是否把已接受的结构化答案写回上下文，并从 `post_apply_plan` 继续。

### 前置条件

- 仓库已存在 AgentSkeleton 上下文
- 用户给出的答案可以被规范化为 response envelope

### 预期宿主行为

- 规范化答案
- 执行 `response --apply`
- 使用 `post_apply_plan` 作为默认继续推进来源
- 进入下一步高价值问题、current priority 或最新 batch 收敛工作

### 失败信号

- 不把已接受答案写回上下文
- 无脑再次执行 `plan`，而不使用 `post_apply_plan`
- apply 之后丢失工作流线程

## 场景 6：Review Candidate 收敛

### 目标

验证宿主是否把 `review_candidates` 当作最新 batch 的临时收敛工作，而不是永久 backlog。

### 前置条件

- 仓库已有已生成文档
- 新 apply 的答案改变了共享仓库语义
- `plan` 或 `post_apply_plan` 中出现 `review_candidates`

### 预期宿主行为

- 识别出之前生成的文档可能需要刷新
- 将 review 集合视为临时的、绑定 change batch 的工作
- 不把它呈现成永久 review 队列

### 失败信号

- 忽略明显相关的 review candidates
- 把 review candidates 当成长期 backlog
- 用无关 review 噪音替代当前应推进的 draft 工作

## 场景 7：过期 Draft Package 刷新

### 目标

验证上下文变化后，宿主是否能刷新 stale 的 `focus-doc` package。

### 前置条件

- 宿主已经拿到过一个 `focus-doc` package
- 新答案被 apply，导致活动 `change_batch_id` 改变
- 宿主又回到同一个 draft 任务

### 预期宿主行为

- 识别旧 package 已过期
- 重新执行 `focus-doc`
- 只从刷新后的 package 继续

### 失败信号

- 继续基于旧 package 起草
- 不比较也不响应 `change_batch_id`
- 基于过期上下文静默产出结果

## 场景 8：Unresolved 状态下的升级

### 目标

验证在自动化不应继续时，宿主是否会停止并升级。

### 前置条件

- response 校验进入 `unresolved`
或
- 根据现有证据，project mode 仍然不明确

### 预期宿主行为

- 暂停自动推进
- 解释唯一阻塞点
- 提出一个有针对性的澄清问题

### 失败信号

- 继续猜测
- 编造缺失业务事实
- 抛出一堆内部工作流细节，而不是聚焦阻塞点

## 场景 9：用户指定某份文档

### 目标

验证宿主是否能在不放弃工作流安全性的前提下，响应用户指定文档的请求。

### 前置条件

- 仓库已存在 AgentSkeleton 上下文
- 用户要求处理某份不一定是 current priority 的文档

### 预期宿主行为

- 执行 `focus-doc --path <document>`
- 检查 readiness 与 missing context
- 只有在安全可写，或允许显式占位起草时才继续

### 失败信号

- 无解释地忽略用户 override
- 不检查 readiness 就直接写指定文档
- 把用户 override 视为可以绕过工作流安全规则的许可

## 场景 10：安静的零命令体验

### 目标

验证整体产品体验，而不只是动作路由正确性。

### 前置条件

- 以真实宿主对话形式运行以上任一场景

### 预期宿主行为

- 大多数情况下隐藏命令名
- 用任务语言解释进展
- 让工作流感觉是被引导的，而不是机械的

### 失败信号

- 反复暴露原始命令
- 让用户自己管理内部状态
- 让体验更像 CLI 教学，而不是宿主接管的工作流

## `M5` 的通过标准

只有在以下条件满足时，`M5` 才应被视为验证通过：

- Codex 通过核心场景
- Claude Code 通过核心场景
- 两类宿主的行为都与共享规范保持一致
- 初始化、起草、apply、收敛、升级等环节都保持零命令体验

## 建议记录的验证证据

每次验证应记录：

- 触发场景的用户提示
- 宿主响应序列
- 宿主选择了哪些 AgentSkeleton 动作
- 宿主是否向用户暴露了命令
- 宿主是否使用了正确的稳定协议字段
- 最终 pass/fail 结论

## 结论

宿主接入验证，不只是验证命令能否执行。

更重要的是验证：宿主能否把 AgentSkeleton 的协议核心转化成一个稳定的零命令产品体验。
