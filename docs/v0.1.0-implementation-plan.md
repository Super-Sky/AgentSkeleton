# v0.1.0 Implementation Plan

## Goal

This plan describes how AgentSkeleton should reach `v0.1.0` as:

- the first public zero-command release for `Codex` and `Claude Code`
- a conversation-driven documentation workflow product
- a host-integrated system where users do not manually operate AgentSkeleton commands

The plan assumes that the CLI/protocol core already exists and that the main remaining work is in host integration, release hardening, and productization.

## Planning Principles

The implementation path should follow these rules:

- keep the CLI core as the workflow engine
- keep natural-language interpretation in host skills/plugins
- optimize for one stable zero-command path before broader extensibility
- favor explicit execution rules over host-specific hidden behavior
- release only after both Codex and Claude Code have a validated main flow

## MVP Definition

`v0.1.0` is complete when the following user experience is real for both supported host families:

1. the user enters a repository and expresses a documentation goal in conversation
2. the host detects that AgentSkeleton should be used
3. the host automatically runs the correct AgentSkeleton steps
4. the host uses AgentSkeleton output to ask follow-up questions or draft a document
5. the host writes accepted structured results back into context
6. the host continues without exposing command-level workflow to the user

## Workstreams

The implementation should be managed across five workstreams:

- host adapter specification
- Codex integration
- Claude Code integration
- protocol stability and test hardening
- release packaging and public documentation

## Milestone Sequence

The recommended sequence is:

1. `M4` Host Adapter Spec
2. `M5` Host Integration Prototypes
3. `M6` Release Hardening

Each milestone should produce user-visible assets, not just internal discussion.

## M4: Host Adapter Spec

### Goal

Turn the current host integration direction into a stable execution specification.

### Why This Comes First

Without a stable spec, Codex and Claude Code integrations will drift and duplicate product logic.

The spec must exist before the official host artifacts are implemented.

### Deliverables

- a host action mapping spec
- a zero-command main-flow spec
- clarification and escalation rules
- stale drafting package rules
- stable protocol field list for `v0.1.0`
- stable versus experimental feature classification

### Required Decisions

The project should explicitly decide:

- how a host detects that a repository should use AgentSkeleton
- when a host should auto-run `init-docs`
- when a host should auto-run `reshape-docs`
- when a host should prefer `plan`
- when a host should prefer `focus-doc`
- when a host should ask a user question instead of drafting
- when `review_candidates` should interrupt forward progress
- when a stale `change_batch_id` requires a new draft package
- which CLI outputs are part of the public contract

### Suggested Artifacts

- `docs/host-action-mapping.md`
- `docs/zero-command-flow.md`
- `docs/protocol-stability.md`

### Exit Criteria

`M4` is complete when:

- the host action rules are documented clearly enough to implement without guesswork
- both Codex and Claude Code integrations can reference the same core spec
- the repository explicitly marks which protocol fields are stable in `v0.1.0`

## M5: Host Integration Prototypes

### Goal

Build and validate one official zero-command integration path for `Codex` and one for `Claude Code`.

### Strategy

The host artifacts should be as thin as possible.

They should:

- interpret user intent
- select AgentSkeleton actions
- pass structured context between host and CLI
- avoid reimplementing AgentSkeleton workflow logic

They should not:

- become a second workflow engine
- invent protocol semantics outside the CLI contract
- hide important product behavior in undocumented prompts

### Deliverables

- one official Codex integration artifact
- one official Claude Code integration artifact
- host-specific enablement docs
- one validated end-to-end zero-command scenario for each host

### Codex Scope

The Codex integration should support:

- repository detection
- automatic initialization versus reshape choice
- `plan` and `focus-doc` driven drafting
- `response --apply` continuation
- stale package refresh behavior

### Claude Code Scope

The Claude Code integration should support:

- repository detection
- automatic initialization versus reshape choice
- `plan` and `focus-doc` driven drafting
- `response --apply` continuation
- stale package refresh behavior

### Suggested Artifact Locations

The exact packaging can vary, but the repository should define one clear home for each host artifact.

Possible approaches include:

- a `skills/` subtree in the repository
- a host-specific integration directory under `docs/` plus companion assets
- a plugin package directory if the host environment supports it cleanly

### Validation Scenarios

Each host should be validated against at least these scenarios:

- new project initialization and first document drafting
- legacy repository reshape and follow-up question flow
- applying new answers and revisiting review candidates
- detecting stale draft packages after context changes

### Exit Criteria

`M5` is complete when:

- each host has one working official integration path
- each host can complete at least one end-to-end zero-command workflow
- the integrations follow the shared host adapter spec rather than custom behavior

## M6: Release Hardening

### Goal

Make the project externally installable, supportable, and releasable as `v0.1.0`.

### Deliverables

- CI for tests and workflow smoke checks
- release packaging strategy
- installation instructions
- host enablement instructions
- updated changelog for `v0.1.0`
- release note draft
- known limitations document

### Required Packaging Decisions

The project should explicitly decide:

- how users install the CLI
- which operating systems are supported at launch
- how release binaries are built and published
- where host integration artifacts are distributed from

### Suggested Release Assets

- GitHub Actions CI workflow
- release build workflow
- installation section in `README.md` and `README.zh-CN.md`
- `CHANGELOG.md` and `CHANGELOG.zh-CN.md` updates
- a `docs/known-limitations.md` file

### Exit Criteria

`M6` is complete when:

- external users can install the CLI without reading internal notes
- external users can enable the Codex integration
- external users can enable the Claude Code integration
- release notes clearly describe what `v0.1.0` supports and what it does not

## Recommended Task Order

The project should execute work in this order:

1. publish the host action mapping and protocol stability docs
2. freeze the `v0.1.0` stable command and field contract
3. build the Codex integration artifact
4. build the Claude Code integration artifact
5. run end-to-end validation scenarios for both hosts
6. add CI and release packaging
7. update README, changelog, and release notes
8. cut `v0.1.0`

This order keeps product decisions ahead of packaging work and avoids polishing a release before the zero-command experience is real.

## Acceptance Checklist

The MVP implementation plan should only be considered complete when all of the following are true:

- the shared host adapter spec is published
- stable `v0.1.0` protocol fields are documented
- one official Codex host artifact exists
- one official Claude Code host artifact exists
- both host artifacts complete a validated zero-command main flow
- CI passes for the CLI core
- release packaging is defined and working
- install and enablement docs are published
- changelog and release notes are ready
- known limitations are documented

## Risk Areas

The main delivery risks are:

- host integrations drifting away from the CLI contract
- unclear escalation rules causing noisy user experience
- stale package handling remaining ambiguous
- release packaging work starting before the host main flow is proven

These risks should be managed by finishing the spec before building host artifacts.

## Non-Goals For v0.1.0

The following items should not block `v0.1.0`:

- broad plugin marketplace distribution
- multiple blueprint families
- advanced customization systems
- rich trace analysis UI
- deeper autonomous repair beyond the current bounded loop

## Conclusion

The fastest path to `v0.1.0` is not adding more CLI breadth.

It is turning the current protocol core into a real zero-command product layer for `Codex` and `Claude Code`, then hardening the release around that proven workflow.
