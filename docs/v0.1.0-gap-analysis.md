# v0.1.0 Gap Analysis

## Target Definition

`v0.1.0` is not just a CLI milestone.

For this project, `v0.1.0` should mean:

- the first public release for zero-command usage inside `Claude Code` and `Codex`
- a conversation-driven workflow where users do not need to memorize or manually invoke AgentSkeleton commands
- a stable enough host integration loop that can be used outside the internal repository context

This is a stronger target than "CLI MVP".

It requires both:

- a working AgentSkeleton core
- at least one usable host-facing integration path per supported host family

## Current Strengths

The repository already has a meaningful core foundation:

- the Go CLI exists and is runnable
- `init-docs`, `reshape-docs`, `plan`, `next`, `prompt`, `response`, `focus-doc`, and `workflow` exist
- the context model, change batch model, review convergence model, and post-apply continuation model are already defined
- host integration intent is documented
- tests for the core package currently pass

This means the project is close to a public CLI/protocol MVP.

It is not yet at a public zero-command host-integrated release.

## Release Blocking Gaps

The following items should be treated as release blockers for `v0.1.0`.

### 1. Official Host Integrations Do Not Yet Exist

The repository documents how hosts should integrate with AgentSkeleton, but it does not yet ship a real official integration artifact for either:

- `Codex`
- `Claude Code`

At least one practical integration path should exist for each host family, such as:

- a reusable skill
- a plugin
- a host wrapper
- a host-side command policy package

If users still need to read the runbook and manually type AgentSkeleton commands, the release target is not met.

### 2. Host Decision Logic Is Documented But Not Productized

The current repository explains the desired flow, but there is not yet a stable, executable host adapter behavior package.

`v0.1.0` needs a clear host-side action mapping for questions such as:

- when to auto-run `init-docs`
- when to auto-run `reshape-docs`
- when to advance `current_priority`
- when to revisit `review_candidates`
- when to re-run `focus-doc` because `change_batch_id` is stale
- when to ask the user a clarification question instead of pushing workflow forward

This logic should not remain only as design prose.

### 3. Zero-Command End-to-End Flow Has Not Been Proven

The core commands work, but the target release is a host experience release.

Before `v0.1.0`, the project should prove at least one real end-to-end path where:

1. the user expresses a goal in conversation
2. the host detects that AgentSkeleton should be used
3. the host invokes the correct CLI steps automatically
4. the host drafts using `focus-doc`
5. the host applies results with `response --apply`
6. the host continues from refreshed state without requiring command-level user intervention

Without this proof, the release would still be a protocol demo rather than a zero-command product.

### 4. Public Release Packaging Is Not Ready

The repository does not yet include visible release packaging infrastructure such as:

- CI automation
- release workflow automation
- binary packaging or install distribution
- documented supported runtime assumptions

An external user should be able to discover:

- how to install AgentSkeleton
- how to enable it in Claude Code
- how to enable it in Codex
- what is officially supported in `v0.1.0`

### 5. Public Version Framing Is Not Yet Updated

The changelog and release framing still reflect an earlier definition-stage state.

Before `v0.1.0`, the project should publish:

- a `v0.1.0` changelog entry
- a release note summary
- explicit known limitations
- a clear distinction between stable and experimental capabilities

## Required Deliverables Before Release

The following deliverables should be considered required.

### Core + Protocol Stability

- confirm the current command surface that is officially stable for `v0.1.0`
- lock the host-consumable structured fields that are part of the public contract
- define which workflow features are stable versus experimental

### Codex Integration

- ship one official Codex integration artifact
- document how the integration detects and invokes AgentSkeleton
- verify that the integration supports the zero-command main flow

### Claude Code Integration

- ship one official Claude Code integration artifact
- document how the integration detects and invokes AgentSkeleton
- verify that the integration supports the zero-command main flow

### Host Adapter Specification

- publish a stable host action mapping spec
- define stale drafting package handling rules
- define escalation and clarification rules
- define continuation rules after `response --apply`

### Release Hardening

- add CI for tests and basic workflow validation
- define release build/distribution method
- add install instructions
- update changelog and release notes

## Recommended Non-Blocking Work

The following items are valuable, but they should not block `v0.1.0` if the zero-command main flow is already solid.

- broader template ecosystem
- richer multi-blueprint support
- deeper auto-repair logic
- more advanced trace inspection UX
- broader plugin marketplace packaging
- nonessential workflow customization layers

## Suggested Milestone Breakdown

To reach `v0.1.0`, the work should be grouped into three milestones.

### M4: Host Adapter Spec

Goal:

- turn the current host integration design into a stable execution spec

Output:

- host action mapping rules
- clarification policy
- stale package policy
- stable protocol field list

### M5: Host Prototypes

Goal:

- prove the zero-command flow in both Codex and Claude Code

Output:

- one Codex integration artifact
- one Claude Code integration artifact
- one validated end-to-end flow for each host

### M6: Release Hardening

Goal:

- make the product externally installable and publicly supportable

Output:

- CI
- release packaging
- install docs
- `v0.1.0` release notes
- support and limitation statement

## Release Readiness Checklist

`v0.1.0` should not be cut until all of the following are true:

- AgentSkeleton core commands pass tests
- the stable command and field contract is documented
- a Codex zero-command integration exists
- a Claude Code zero-command integration exists
- the host action mapping is documented as an execution spec
- at least one end-to-end zero-command flow is validated for each host
- installation and enablement documentation exists
- changelog and release notes are prepared
- known limitations are explicitly documented

## Conclusion

The project is already close to a useful protocol-core release.

The main remaining gap to `v0.1.0` is not the CLI engine itself.

It is the host product layer:

- official integrations
- stable host behavior rules
- validated zero-command experience
- public release packaging

If those pieces are completed, `v0.1.0` can credibly be presented as the first public zero-command AgentSkeleton release for Claude Code and Codex.
