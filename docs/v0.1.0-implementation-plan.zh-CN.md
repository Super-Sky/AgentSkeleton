# v0.1.0 实施计划

## 目标

本文档说明 AgentSkeleton 应如何达到 `v0.1.0`，其目标是：

- 面向 `Codex` 与 `Claude Code` 的第一个零命令感知公开版本
- 一个由对话驱动的文档工作流产品
- 一个用户无需手工操作 AgentSkeleton 命令的宿主集成系统

本计划的前提是：CLI / protocol core 已基本存在，剩余主要工作集中在宿主接入、发布加固和产品化。

## 规划原则

实施路径应遵循以下原则：

- 保持 CLI core 作为工作流引擎
- 保持自然语言理解留在宿主 skill/plugin 中
- 先打通一条稳定的零命令主链路，再考虑更广泛扩展
- 用显式执行规则替代宿主侧隐藏行为
- 只有在 Codex 与 Claude Code 都验证了主链路后才发布

## MVP 定义

当以下体验在两类宿主上都真实存在时，`v0.1.0` 才算完成：

1. 用户进入仓库后，通过对话表达文档目标
2. 宿主识别出当前应启用 AgentSkeleton
3. 宿主自动执行正确的 AgentSkeleton 步骤
4. 宿主基于 AgentSkeleton 输出继续追问或起草文档
5. 宿主把被接受的结构化结果写回上下文
6. 宿主持续推进，而不把命令级流程暴露给用户

## 工作流分组

建议把实施工作拆成五条工作流：

- host adapter 规范
- Codex 接入
- Claude Code 接入
- 协议稳定性与测试加固
- 发布打包与对外文档

## 里程碑顺序

推荐顺序是：

1. `M4` Host Adapter Spec
2. `M5` Host Integration Prototypes
3. `M6` Release Hardening

每个里程碑都应产出可见资产，而不只是内部讨论结果。

## M4: Host Adapter Spec

### 目标

把当前宿主接入方向收敛成一套稳定的执行规范。

### 为什么先做这个

如果没有稳定规范，Codex 和 Claude Code 的接入实现很容易漂移，并开始重复产品逻辑。

因此，官方宿主接入产物必须建立在统一规范之上。

### 交付物

- 一份 host action mapping spec
- 一份 zero-command main-flow spec
- clarification 与 escalation 规则
- stale drafting package 规则
- `v0.1.0` 的稳定协议字段清单
- stable 与 experimental 能力边界

### 必须明确的决策

项目应明确以下问题：

- 宿主如何识别当前仓库应启用 AgentSkeleton
- 什么时候自动执行 `init-docs`
- 什么时候自动执行 `reshape-docs`
- 什么时候优先使用 `plan`
- 什么时候优先使用 `focus-doc`
- 什么时候应先向用户追问，而不是直接起草
- 什么时候 `review_candidates` 应打断向前推进
- 什么时候 stale `change_batch_id` 必须触发新的 draft package
- 哪些 CLI 输出字段属于公开稳定契约

### 建议产物

- `docs/host-action-mapping.md`
- `docs/zero-command-flow.md`
- `docs/protocol-stability.md`

### 退出标准

当满足以下条件时，`M4` 才算完成：

- 宿主动作规则已经清楚到无需猜测即可实现
- Codex 与 Claude Code 都可以引用同一套核心规范
- 仓库已经明确标记 `v0.1.0` 中哪些协议字段是稳定的

## M5: Host Integration Prototypes

### 目标

为 `Codex` 和 `Claude Code` 各实现一条官方零命令接入路径，并完成验证。

### 策略

宿主接入产物应尽可能薄。

它们应负责：

- 理解用户意图
- 选择 AgentSkeleton 动作
- 在宿主与 CLI 之间传递结构化上下文
- 避免重复实现 AgentSkeleton 工作流逻辑

它们不应负责：

- 变成第二套工作流引擎
- 在 CLI 契约之外自定义协议语义
- 把关键产品行为藏进未文档化的 prompt

### 交付物

- 一个官方 Codex 接入产物
- 一个官方 Claude Code 接入产物
- 宿主侧启用文档
- 每类宿主至少一条经过验证的零命令端到端链路

### Codex 范围

Codex 接入应支持：

- 仓库识别
- 自动判断初始化还是 reshape
- 基于 `plan` 与 `focus-doc` 的起草推进
- `response --apply` 之后的继续推进
- stale package 刷新行为

### Claude Code 范围

Claude Code 接入应支持：

- 仓库识别
- 自动判断初始化还是 reshape
- 基于 `plan` 与 `focus-doc` 的起草推进
- `response --apply` 之后的继续推进
- stale package 刷新行为

### 建议接入产物位置

具体封装方式可以灵活，但仓库应为每类宿主明确一个清晰的归属位置。

可选方式包括：

- 在仓库中增加 `skills/` 子目录
- 在 `docs/` 下增加宿主接入目录并配套资产
- 如果宿主环境成熟支持，也可引入 plugin package 目录

### 验证场景

每类宿主至少应验证以下场景：

- 新项目初始化与第一份文档起草
- 老项目 reshape 与后续追问流程
- 应用新答案后回看 `review_candidates`
- 上下文变化后识别 stale draft package

### 退出标准

当满足以下条件时，`M5` 才算完成：

- 每类宿主都有一条官方、可工作的接入路径
- 每类宿主都能完成至少一条端到端零命令工作流
- 宿主接入实现遵循共享 host adapter 规范，而不是各写一套行为

## M6: Release Hardening

### 目标

让项目具备可安装、可支持、可作为 `v0.1.0` 正式发布的状态。

### 交付物

- 为测试和工作流 smoke check 提供 CI
- release packaging 策略
- 安装说明
- 宿主启用说明
- `v0.1.0` changelog 更新
- release notes 草稿
- 已知限制文档

### 必须明确的打包决策

项目应明确：

- 用户如何安装 CLI
- 首发支持哪些操作系统
- release 二进制如何构建和发布
- 宿主接入产物从哪里分发

### 建议的发布资产

- GitHub Actions CI workflow
- release build workflow
- `README.md` 与 `README.zh-CN.md` 中的安装章节
- `CHANGELOG.md` 与 `CHANGELOG.zh-CN.md` 更新
- `docs/known-limitations.md`

### 退出标准

当满足以下条件时，`M6` 才算完成：

- 外部用户无需阅读内部说明就能安装 CLI
- 外部用户可以启用 Codex 接入
- 外部用户可以启用 Claude Code 接入
- release notes 清楚说明 `v0.1.0` 支持什么、不支持什么

## 建议任务顺序

建议按以下顺序执行：

1. 发布 host action mapping 与 protocol stability 文档
2. 冻结 `v0.1.0` 的稳定命令面与字段契约
3. 构建 Codex 接入产物
4. 构建 Claude Code 接入产物
5. 为两类宿主跑通端到端验证场景
6. 补齐 CI 与 release packaging
7. 更新 README、changelog 与 release notes
8. 切出 `v0.1.0`

这个顺序能确保先做产品决策，再做打包工作，避免在零命令体验尚未成立之前就过早进入发布打磨阶段。

## 验收清单

只有当以下条件全部成立时，这份 MVP 实施计划才算真正完成：

- 共享 host adapter 规范已发布
- `v0.1.0` 的稳定协议字段已文档化
- 存在一个官方 Codex 接入产物
- 存在一个官方 Claude Code 接入产物
- 两类宿主都完成了一条经过验证的零命令主链路
- CLI core 的 CI 测试通过
- release packaging 已定义并可工作
- 安装与启用文档已发布
- changelog 与 release notes 已准备
- 已知限制已文档化

## 风险点

主要交付风险包括：

- 宿主接入实现偏离 CLI 契约
- 升级/澄清规则不清晰，导致用户体验噪音过高
- stale package 处理持续含糊
- 在宿主主链路尚未验证前就提前投入发布打包

这些风险应通过“先完成规范，再实现宿主接入”的顺序来控制。

## v0.1.0 的非目标

以下内容不应阻塞 `v0.1.0`：

- 广泛的插件市场分发
- 多套 blueprint 家族
- 高级定制系统
- 丰富的 trace 分析 UI
- 超出当前有界循环之外的更深层自动修复

## 结论

通往 `v0.1.0` 的最快路径，不是继续扩展 CLI 宽度。

而是把当前的 protocol core 变成面向 `Codex` 与 `Claude Code` 的真实零命令产品层，然后围绕这条已验证主链路完成发布加固。
