# Reference Gap Analysis

## Purpose

This document compares AgentSkeleton against `secure_digital_platform` as a reference sample and clarifies:

- which AI-friendly assets already exist in the sample
- which parts AgentSkeleton has already abstracted
- which important capabilities are still missing

This document focuses on reusable documentation and collaboration patterns only. It does not copy business implementation details or private project intelligence.

## Reference Sample Observations

The reference project already contains several mature assets:

- repository entry documents
  - `README.md`
  - `AGENTS.md`
  - `CLAUDE.md`
- method-oriented documentation assets
  - `docs/README.md`
  - `docs/CODE_STYLE_GUIDE.md`
  - `docs/DOC_TEMPLATES.md`
  - `docs/REPO_WORKFLOW_GUIDE.md`
  - `docs/TASK_DELIVERY_GUIDE.md`
  - `docs/TASK_EXECUTION_TEMPLATE.md`
- versioned documentation directories
  - `docs/vX.Y.Z/README.md`
  - `docs/vX.Y.Z/features/...`
- dual-host skill placement
  - `.agents/skills/`
  - `.claude/skills/`
- explicit README guidance for project status, historical structure, skill placement, and document responsibilities

This means the reference project has already evolved from a plain code repository into a repository that is understandable and collaborative for agents.

## Covered Capabilities

AgentSkeleton already covers these areas:

- bilingual repository entry and product documents
- dual-host collaboration entry points through `AGENTS.md` and `CLAUDE.md`
- documentation blueprints and base templates
- versioned documentation blueprints
- legacy documentation reshape blueprints
- CLI context model
- minimal `plan` / `next` / `prompt` / `response` / `workflow` loop
- model-response validation, retry handling, and repair prompts
- `<output-dir>/.agentskeleton` process artifact model
- workflow trace persistence

These capabilities are enough to support the minimum product shape for a conversation-driven documentation guidance tool.

## Remaining Gaps

### 1. The document generation loop is still incomplete

Even though `workflow --write-plan-files` exists, it is still basic:

- only part of the core documents are covered
- there is no full per-document progression state machine yet
- tracking for generated documents, pending documents, and missing context is still too coarse

This is the highest-priority gap.

### 2. Method-oriented documentation assets are not fully productized

We already abstracted blueprint-level assets such as `CODE_STYLE_GUIDE`, `DOC_TEMPLATES`, `REPO_WORKFLOW_GUIDE`, `TASK_DELIVERY_GUIDE`, and `TASK_EXECUTION_TEMPLATE`, but we still lack:

- which method documents should be generated by default for which project types
- how priorities differ between new-project and legacy-project flows
- how the CLI should recommend those documents based on current context

In other words, the templates exist, but the scheduling logic is still weak.

### 3. The skill integration layer is still underdefined

The reference project already has a stable dual-placement model:

- `.agents/skills/`
- `.claude/skills/`

AgentSkeleton already states at the principle level that skills are external capabilities and should be created by standard tooling, but it still does not fully define:

- which kinds of projects should include skills
- when skills should enter the generation plan
- how documentation blueprints and skill placement should coordinate

### 4. Versioned documentation is not tightly connected to delivery flow yet

The reference project is already using versioned docs in real iteration work. AgentSkeleton currently has the blueprints, but still lacks:

- how a specific iteration enters `docs/vX.Y.Z/`
- when feature documents should be created
- when review checklists should appear
- how those outputs should connect to workflow state

### 5. There is no formal mapping document from sample assets to generic abstractions

We currently know which assets exist in the reference project, but we do not yet have a formal document that explains:

- which sample asset maps to which AgentSkeleton blueprint category
- which parts were intentionally excluded
- why they were excluded

Without this, future abstraction work can drift into repeated debates.

## What Should Not Be Copied

The following should not be directly templated into AgentSkeleton:

- the reference project's concrete business entities
- the reference project's API paths and service decomposition
- detailed historical compatibility specifics
- the reference project's exact code layering implementation
- any real data, private fields, or business intelligence

AgentSkeleton should abstract how a repository explains itself and how documentation collaboration becomes stable, not how one business project is implemented.

## Recommended Next Steps

Suggested order of work:

1. Make `workflow` output the current priority document and the context required to generate it.
2. Write generated-document state back into context so the documentation loop becomes durable.
3. Define generation priority rules for method-oriented documents across project modes.
4. Define skill integration trigger points instead of leaving skills only at the principles layer.
5. Add a formal sample-to-blueprint mapping document.

## Current Assessment

The conclusion is:

AgentSkeleton is moving in the same direction as the documentation-mature form of `secure_digital_platform`, but they are not the same product.

- `secure_digital_platform` is a mature practice sample inside a business repository
- AgentSkeleton is the product that abstracts those practices into a generic CLI and documentation blueprint system

So AgentSkeleton should learn from how that repository matured its documentation and collaboration assets, not copy its business structure.
