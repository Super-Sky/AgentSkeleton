# 零命令主链路

## 目的

本文档用于定义 AgentSkeleton 在 `Codex` 与 `Claude Code` 中的目标零命令工作流。

它描述了 `v0.1.0` 应提供的用户体验，以及宿主为了让这条体验稳定成立所必须承担的最小行为。

## 体验目标

用户应把 AgentSkeleton 体验成一种原生于对话的工作流。

用户不应需要：

- 记忆 AgentSkeleton 命令名
- 自己决定下一步运行哪个 CLI 步骤
- 手工维护工作流状态
- 手工判断 drafting package 是否过期
- 手工追踪已生成文档之间的收敛关系

这些责任应由宿主接入层吸收。

## 核心用户故事

默认的零命令故事应是：

1. 用户通过对话描述一个文档目标
2. 宿主识别出当前应启用 AgentSkeleton
3. 宿主自动初始化或恢复 AgentSkeleton
4. 宿主只追问最高价值的澄清问题
5. 宿主使用 AgentSkeleton 上下文起草或更新下一份文档
6. 宿主把被接受的结构化进展写回工作流状态
7. 宿主持续循环，直到当前文档目标满足为止

## 范围

本文档覆盖：

- 零命令主循环
- 初始化行为
- 起草行为
- 澄清行为
- 收敛行为
- 上下文更新后的继续推进规则

本文档不覆盖：

- plugin 打包
- 安装说明
- 宿主 skill 的内部实现细节

## 流程状态

宿主应使用以下状态来理解零命令主链路：

- activation
- initialization
- planning
- clarification
- drafting
- applying
- convergence
- escalation

这些是宿主行为状态，不一定与 CLI 命令一一对应。

## 状态 1：Activation

### 触发条件

用户进入仓库对话，并表达了文档化或仓库结构相关目标。

### 宿主职责

- 判断 AgentSkeleton 是否已在仓库中启用
- 判断当前任务是否应启用 AgentSkeleton
- 判断这个请求是否适合留在 AgentSkeleton 范围内

### 预期用户体验

用户只是在描述目标。

宿主不要求用户去选择命令。

## 状态 2：Initialization

### 触发条件

仓库中尚未初始化 AgentSkeleton。

### 宿主职责

- 判断仓库应使用 `init-docs` 还是 `reshape-docs`
- 如果模式有歧义，则追问一个澄清问题
- 执行选定的初始化动作
- 建立第一份工作流上下文

### 预期用户体验

用户应把这一步体验成由宿主处理的初始化过程。

宿主可以追问一个高价值问题，例如“这是新项目还是老仓库”，但不应把初始化变成命令教学。

## 状态 3：Planning

### 触发条件

上下文已经存在，宿主需要获取当前工作流快照。

### 宿主职责

- 执行 `plan`
- 检查 `current_priority`
- 检查 `missing_information`
- 检查 `review_candidates`
- 判断下一步最优动作是澄清、起草还是收敛

### 预期用户体验

宿主应直接给出清晰下一步，而不是把底层规划机制暴露给用户。

## 状态 4：Clarification

### 触发条件

由于 required context 缺失或路由歧义，当前还无法可靠起草。

### 宿主职责

- 只追问下一个最高价值问题
- 保持问题狭窄，并与当前工作流需求直接相关
- 当缺失上下文已经明确时，避免进入宽泛探索式提问
- 把被接受的答案规范化成 response envelope

### 预期用户体验

宿主提出的追问应显得必要、聚焦、贴近上下文。

用户不应感觉自己在机械填表。

## 状态 5：Drafting

### 触发条件

当前 priority 文档，或用户指定文档，已经具备起草条件，或在显式占位的前提下可以起草。

### 宿主职责

- 执行 `focus-doc`
- 消费 drafting package
- 使用 `required_context`、`missing_context`、`available_context` 与 `suggested_sections`
- 在不编造缺失事实的前提下起草或更新目标文档

### 预期用户体验

宿主应自然地继续推进工作，而不是询问用户要运行什么命令或检查哪个数据结构。

## 状态 6：Applying

### 触发条件

宿主已经获得被接受的结构化答案，或完成了需要写回上下文的重要工作流进展。

### 宿主职责

- 把结构化答案规范化成 response envelope
- 执行 `response --apply`
- 消费 `post_apply_plan`
- 从返回的工作流状态继续推进

### 预期用户体验

用户应体验到一种自然的前进感，而不需要手工同步内部状态。

## 状态 7：Convergence

### 触发条件

最新 change batch 已经影响到之前生成的文档。

### 宿主职责

- 检查 `review_candidates` 或 `review_after_draft`
- 判断收敛工作应立即处理，还是在当前 draft 完成后处理
- 把收敛视为绑定最新 batch 的临时工作，而不是永久 backlog

### 预期用户体验

宿主可以告知用户：因为新信息改变了共享语义，所以某份旧文档需要刷新。

用户不应自己维护“哪些文档被影响”的清单。

## 状态 8：Escalation

### 触发条件

工作流无法安全自动继续。

### 宿主职责

- 停止自动推进
- 解释唯一阻塞点
- 提出一个针对性的澄清问题

### 升级示例

- project mode 不明确
- response 校验最终进入 `unresolved`
- 指定起草会依赖编造事实
- 当前请求与活动工作流状态冲突

### 预期用户体验

升级应像一次谨慎暂停，而不是系统失败转储。

## 标准主循环

推荐的零命令主循环应是：

1. 当仓库与任务匹配时激活 AgentSkeleton
2. 如有需要，用 `init-docs` 或 `reshape-docs` 初始化
3. 执行 `plan`
4. 如果起草被阻塞，则追问最高价值问题
5. 通过 `response --apply` 写回被接受的结构化答案
6. 从 `post_apply_plan` 继续
7. 当起草条件满足时，执行 `focus-doc`
8. 起草目标文档
9. 写回新的结构化进展
10. 只有在需要时才回看最新 batch 的收敛工作

这个循环应持续推进，直到当前文档目标满足。

## 起草包新鲜度

宿主必须把 `focus-doc` 返回的 package 视为临时产物。

当满足以下条件时，应刷新 package：

- `change_batch_id` 已不是当前最新值
- package 生成后又有新答案被 apply
- 宿主切换任务后又返回同一个 draft

宿主不应基于过期 package 继续起草。

## 对话风格要求

宿主应：

- 使用任务级语言交流
- 尽量减少问题数量
- 用用户语言解释下一步
- 除非用户明确要求，否则避免引用内部协议细节

宿主不应：

- 默认暴露底层 CLI 顺序
- 要求用户手工检查 workflow 字段
- 让用户自己管理 change batch 或 drafting package 新鲜度

## v0.1.0 成功标准

当 `Codex` 与 `Claude Code` 都能完成以下动作时，零命令主链路才算真实成立：

- 无需用户选择命令即可激活 AgentSkeleton
- 自动初始化或恢复上下文
- 在澄清与起草之间正确路由
- 使用 `focus-doc` 作为起草事实来源
- 使用 `response --apply` 写回结构化进展
- 从 `post_apply_plan` 继续推进
- 仅在最新 change batch 需要时回看收敛工作
- 在缺失事实时停止并升级，而不是编造

## 非目标

对 `v0.1.0` 来说，零命令主链路不需要包含：

- 完全无人反馈下的全仓库自动完成
- 高级多文档并行规划
- 广泛可定制的对话策略
- 完整的插件市场生态支持

## 结论

零命令主链路是 `v0.1.0` 面向用户的核心承诺。

CLI core 为这个承诺提供基础，而宿主接入层必须把它实现成自然、安静、可靠的体验。
