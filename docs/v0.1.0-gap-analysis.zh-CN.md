# v0.1.0 差距分析

## 目标定义

`v0.1.0` 不是一个单纯的 CLI 里程碑。

对本项目来说，`v0.1.0` 应当意味着：

- 第一个面向 `Claude Code` 与 `Codex` 的零命令感知公开版本
- 一个完全由对话驱动的工作流，用户不需要记忆或手工调用 AgentSkeleton 命令
- 一条足够稳定的宿主接入主链路，能够脱离内部仓库环境供外部用户使用

这个目标比“CLI MVP”更强。

它同时要求：

- AgentSkeleton core 可用
- 每类受支持宿主至少有一条可用的官方接入路径

## 当前已有基础

仓库已经具备一套有意义的 core 基础：

- Go CLI 已存在且可运行
- `init-docs`、`reshape-docs`、`plan`、`next`、`prompt`、`response`、`focus-doc`、`workflow` 已经实现
- context model、change batch model、review convergence model、post-apply continuation model 都已经定义
- 宿主接入意图已经有文档说明
- core package 测试当前可以通过

这意味着项目已经接近一个公开可用的 CLI / protocol MVP。

但它还没有达到“公开可用的零命令宿主集成版本”。

## 发布阻塞项

以下内容应视为 `v0.1.0` 的发布阻塞项。

### 1. 官方宿主接入物尚不存在

仓库已经描述了宿主应如何接入 AgentSkeleton，但目前还没有真正随仓库交付的官方接入产物，至少对以下宿主都是如此：

- `Codex`
- `Claude Code`

每类宿主至少应存在一种实际可用的接入方式，例如：

- 可复用 skill
- plugin
- host wrapper
- 宿主侧命令策略包

如果用户仍然需要自己阅读 runbook 并手动输入 AgentSkeleton 命令，那么发布目标就还没有达成。

### 2. 宿主决策逻辑仍停留在文档层

当前仓库解释了期望流程，但还没有一套稳定、可执行的 host adapter 行为包。

`v0.1.0` 需要把以下问题整理成明确的宿主动作映射：

- 什么时候自动执行 `init-docs`
- 什么时候自动执行 `reshape-docs`
- 什么时候推进 `current_priority`
- 什么时候回看 `review_candidates`
- 什么时候因为 `change_batch_id` 过期而重新执行 `focus-doc`
- 什么时候应先向用户追问，而不是继续推进工作流

这些逻辑不能只停留在设计文档里。

### 3. 零命令端到端链路尚未被验证

当前核心命令已经可用，但目标版本是“宿主体验版”，不是“协议演示版”。

在 `v0.1.0` 之前，项目至少应证明一条真实端到端链路，其中：

1. 用户通过对话表达目标
2. 宿主识别出当前应启用 AgentSkeleton
3. 宿主自动调用正确的 CLI 步骤
4. 宿主使用 `focus-doc` 进行起草
5. 宿主通过 `response --apply` 写回结果
6. 宿主基于刷新后的状态继续推进，而不是要求用户进行命令级操作

如果这条链路没有跑通，发布出来的仍然更像协议样品，而不是零命令产品。

### 4. 面向公开发布的打包与交付尚未就绪

仓库当前还没有明显的发布基础设施，例如：

- CI 自动化
- release workflow 自动化
- 二进制打包或安装分发方式
- 对外声明的运行环境支持假设

外部用户应能明确知道：

- 如何安装 AgentSkeleton
- 如何在 Claude Code 中启用它
- 如何在 Codex 中启用它
- `v0.1.0` 正式支持哪些能力

### 5. 对外版本 framing 仍停留在更早阶段

当前 changelog 与 release framing 仍然带有较强的“定义阶段”色彩。

在 `v0.1.0` 之前，项目应补齐：

- 一条 `v0.1.0` changelog 记录
- 一份 release note 摘要
- 显式的已知限制说明
- 对 stable 与 experimental 能力的边界说明

## 发布前必须交付的内容

以下交付物应视为必需项。

### Core 与协议稳定性

- 确认当前哪些命令面属于 `v0.1.0` 的正式稳定能力
- 锁定哪些结构化字段属于公开协议契约
- 定义哪些 workflow 能力属于 stable，哪些属于 experimental

### Codex 接入

- 提供一个官方 Codex 接入产物
- 说明该接入层如何识别并调用 AgentSkeleton
- 验证它支持零命令主链路

### Claude Code 接入

- 提供一个官方 Claude Code 接入产物
- 说明该接入层如何识别并调用 AgentSkeleton
- 验证它支持零命令主链路

### Host Adapter 规范

- 发布一份稳定的 host action mapping spec
- 定义起草包过期处理规则
- 定义升级为追问/澄清的规则
- 定义 `response --apply` 之后的继续推进规则

### 发布加固

- 为测试和基础工作流校验补齐 CI
- 定义 release 构建与分发方式
- 补充安装说明
- 更新 changelog 与 release notes

## 建议但不阻塞的工作

以下内容有价值，但如果零命令主链路已经稳定，则不应阻塞 `v0.1.0`：

- 更广泛的模板生态
- 更丰富的多蓝图支持
- 更深入的 auto-repair 逻辑
- 更高级的 trace 检查体验
- 更广泛的插件市场分发封装
- 非必要的 workflow 定制层

## 建议的里程碑拆分

为了达到 `v0.1.0`，建议把工作拆成三个里程碑。

### M4: Host Adapter Spec

目标：

- 把当前的宿主接入设计收敛成稳定执行规范

产出：

- host action mapping 规则
- clarification policy
- stale package policy
- stable protocol field list

### M5: Host Prototypes

目标：

- 在 Codex 与 Claude Code 中证明零命令主链路

产出：

- 一个 Codex 接入产物
- 一个 Claude Code 接入产物
- 每个宿主至少一条经过验证的端到端链路

### M6: Release Hardening

目标：

- 让产品具备可安装、可对外支持、可正式发布的状态

产出：

- CI
- release packaging
- install docs
- `v0.1.0` release notes
- support 与 limitation 声明

## 发布就绪检查表

在以下条件全部满足之前，不应切出 `v0.1.0`：

- AgentSkeleton core commands 测试通过
- 稳定命令面与结构化字段契约已文档化
- 存在一个 Codex 零命令接入产物
- 存在一个 Claude Code 零命令接入产物
- host action mapping 已作为执行规范文档化
- 每类宿主至少验证一条端到端零命令链路
- 已存在安装与启用文档
- changelog 与 release notes 已准备
- 已知限制已显式文档化

## 结论

项目已经接近一个可用的 protocol-core 发布版本。

距离 `v0.1.0` 的主要差距，并不在 CLI 引擎本身。

真正还缺的是宿主产品层：

- 官方接入物
- 稳定的宿主行为规则
- 经过验证的零命令体验
- 面向公开发布的打包与交付

只有这些部分补齐后，`v0.1.0` 才能有说服力地被定义为面向 Claude Code 与 Codex 的第一个零命令公开版。
